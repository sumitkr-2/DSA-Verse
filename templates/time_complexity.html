Time Complexity

What's Time Complexity & Why Do We Need It?

Let's try to understand this by taking an example of a civil engineer. When a civil engineer does a survey, they need to take care of certain standard parameters. Suppose they are asked to tell whether the given cement mixture being used at a construction site is good or not. They will check parameters like water ratio, aggregate ratio, sand ratio, etc.

But when we talk about a Computer Science Engineer, their job is to do a survey on the code. So, if a software engineer is asked whether a given piece of code is good or not, how would they tell? The answer to this question is Time and Space Complexity.

Definition: Time complexity of an algorithm quantifies the amount of time taken by an algorithm to run as a function of the length of the input. Similarly, Space complexity quantifies the amount of space or memory taken by an algorithm to run as a function of the length of the input.

Asymptotic Notations

We use three main asymptotic notations to describe the running time of an algorithm.

1. Big-Θ (Big-Theta) notation

Let's look at a simple implementation of linear search: 

var doLinearSearch = function(array, targetValue) {
  for (var guess = 0; guess < array.length; guess++) {
    if (array[guess] === targetValue) { 
        return guess;  // found it!
    }
  }
  return -1;  // didn't find it
};

Let's denote the size of the array (array.length) by n. The maximum number of times the for-loop can run is n, and this worst case occurs when the value being searched for is not in the array. Each time the loop iterates, it does a constant amount of work. Therefore, the total time for linear search in the worst case is c₁⋅n + c₂.

When we use Big-Θ notation, we are saying that we have an asymptotically tight bound on the running time. We drop constant factors and lower-order terms. For linear search, we say the worst-case running time is Θ(n). This means that for a large enough n, the running time is sandwiched between k₁⋅n and k₂⋅n.

2. Big-O (Big-Oh) Notation

Sometimes, we only want to bound the running time from above. We use Big-O notation for this. If a running time is O(f(n)), then for a large enough n, the running time is at most k⋅f(n) for some constant k. This gives an asymptotic upper bound.

For example, the running time of binary search is always O(log n) because it never performs worse than logarithmic time. However, in the best case, it can be Θ(1). Since O(log n) covers all cases, it's a correct (though sometimes not tight) bound.

3. Big-Ω (Big-Omega) Notation

When we want to say that an algorithm takes at least a certain amount of time, we use Big-Ω notation. If a running time is Ω(f(n)), then for a large enough n, the running time is at least k⋅f(n). This provides an asymptotic lower bound.

For example, any algorithm that must look at every element in an array of size n will have a running time of at least Ω(n). This notation is useful for proving that an algorithm is optimal.
